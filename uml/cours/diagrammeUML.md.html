Les diagrammes utiles pour modéliser une application web

Nous utiliserons 3 catégories de diagrammes : * un diagramme décrivant les fonctionnalités * un diagramme décrivant les données, la partie invariante * des diagrammes décrivant les actions dans le temps

Description des fonctionnalités

Un seul diagramme. C’est le diagramme des cas d’utilisation (USE CASE) L’équivalent en SCRUM est le USER STORY. Avec ce diagramme, on identifie les acteurs et leur rôles ainsi que les fonctionnalités de l’application.
On définit un Use Case global pour l’application, puis des Uses Cases détaillés si besoin pour chaqu’un des Uses case principaux.
exemple

Chaque use case doit donner lieu à la description d’un scénario textuel décrivant tout le cheminement du traitement : * quand tout se passe bien * quand quelque chose se passe mal Les scénaris permettent aussi de décrire les champs impactées par le traitement. On le relie souvent à une page.

Description des données

Un seul diagramme. C’est le diagramme des classes (MOO) On définit les classes (entités) qui sont persistantes (celles qui seront stockées en base de données).

Description des traitements

Ici nous avons 3 diagrammes. Vous utilisez au choix le diagramme des séquences ou le diagramme d’activité.
Pour l’enchaînement des écrans, on utilise le diagramme de navigation. C’est ce que l’on appelle la cinématique.

Le diagramme d’activité

Le diagramme d’activité décrit un comportement en fonction du temps. Il ressemble beaucoup au flowchart (diagramme de programmation) vu avec Flowgorithm.

exemples

Le diagramme de navigation

Le diagramme de navigation définit l’enchaînement des vues (écrans). Il est complémentaire du diagramme d’activité.

Le diagramme des séquences SI

On définit une action utilisateur dans le temps.
Une action exécute une URL donc un contrôleur, ce qui déclenche une action et renvoie un réponse. Bien utile dans une architecture MVC, car il aide à la réalisation des contrôleurs. Avec ce diagramme, nous sommes capable de nommer un requête http.
